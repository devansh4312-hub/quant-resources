4-Week Roadmap

Striver A2Z + CSES + CF

Activity

Weekly Game Plan

What to do each week
W1

Foundations & STL Basics

~2–3 hrs/day

🎯 Goal

Get comfortable with C++ competitive programming workflow, understand time complexity so you can estimate whether your solution will pass, and build fluency with basic STL containers. By the end of this week, you should be solving Codeforces 800-rated problems in under 15 minutes.

📅 Day-by-Day

Day 1–2Learn Big O notation. Watch Striver's time complexity video, then read Chapter 2 of the CP Handbook. Practice estimating: given N ≤ 10⁶, can an O(n²) solution pass? (No — you need O(n log n) or better.) Solve CSES "Weird Algorithm" and "Missing Number" to get your CPH + Competitive Companion workflow running smoothly.
Day 3–4Basic math: implement Euclidean GCD, understand LCM = a/gcd(a,b)*b (multiply before divide to avoid overflow). Do modular arithmetic basics — why (a+b)%m = ((a%m)+(b%m))%m matters. Solve CSES "Repetitions" and "Increasing Array". Start Striver's recursion playlist — factorial, fibonacci, print 1 to N.
Day 5–6Deep-dive into STL: vectors (push_back, sort, resize), strings (substr, find), pairs, and range-based for loops. Write small test programs to build muscle memory. Solve 3–4 Codeforces 800 problems from recent Div 4 contests (just A and B problems). Focus on reading the problem carefully — most 800-rated problems are about implementation, not algorithms.
Day 7Recursion practice: generate all subsequences of a string, power(a,b) using recursion. Review the week — revisit any concept that felt shaky. Solve 2 more CF 800 problems as a cooldown.
💡 Tip: Don't overthink problem difficulty yet. Your main enemy this week is unfamiliarity with the workflow — reading from stdin, printing to stdout, submitting, handling edge cases. Get those reps in. Every problem you solve, even trivial ones, builds speed.
W2

Sorting, Searching & Advanced STL

~3 hrs/day

🎯 Goal

Master sorting with custom comparators, learn binary search (both manual and STL), and get comfortable with maps, sets, and priority queues. These are the bread and butter of competitive programming — you'll use them in almost every contest. Target: Codeforces 900-rated problems.

📅 Day-by-Day

Day 1–2Sorting: understand std::sort, write custom comparators (sort by second element of pairs, sort strings by length). Practice: sort an array of events by end time (this is the foundation of greedy scheduling). Solve CSES "Distinct Numbers" and "Apartments". Read CP Handbook Chapter 3 (Sorting).
Day 3–4Binary Search: implement it manually (the classic while(lo <= hi) pattern), then learn std::lower_bound and upper_bound. Key insight — binary search works whenever you have a monotonic condition. Solve CSES "Ferris Wheel" and start "Subarray Sums I". Do 2–3 CF 900 problems that involve sorting + greedy.
Day 5–6Advanced STL day: maps (frequency counting — this is huge), sets (ordered unique elements, lower_bound on sets), multisets, priority queues (min-heap: priority_queue<int, vector<int>, greater<int>>). Practice: solve "find the k-th smallest element" using a set, "find top 3 frequent elements" using a map + sorting. Solve 3 CF Div 3/4 B/C problems.
Day 7Do a virtual contest: pick a recent Codeforces Div 4 and try it under the 2-hour time limit. Don't worry about your score — just experience the pressure. After the contest, upsolve every problem you couldn't finish. This is the most important learning step.
💡 Tip: When you see "find something optimal" in a problem statement, think sorting first. When you see "find if X exists" or "find minimum/maximum that satisfies a condition", think binary search. These two patterns cover a shocking number of problems at this level.
W3

Number Theory & Sliding Window

~3–4 hrs/day

🎯 Goal

This is where you start feeling like a real competitive programmer. Number theory (modular exponentiation, sieve) unlocks a category of math-heavy problems. Two pointers and sliding window are your first "clever" techniques — they turn O(n²) brute forces into O(n) solutions. Target: CF 1000-rated problems.

📅 Day-by-Day

Day 1–2Number theory block: implement binary exponentiation (a^b mod m in O(log b)) — you'll use this constantly. Learn the Sieve of Eratosthenes to precompute primes up to 10⁷. Practice prime factorization. Solve CSES "Exponentiation". Read CP Handbook Chapter 21 (Number Theory).
Day 3Bit manipulation: understand AND, OR, XOR, left/right shifts. Key tricks — check if i-th bit is set (n & (1 << i)), count set bits (__builtin_popcount), check if power of 2 (n & (n-1) == 0). These are quick wins that show up as sub-problems in contests. Practice with 2–3 easy bit problems from Striver's sheet.
Day 4–5Two Pointers: the classic pattern — one pointer at start, one at end, move inward. Use cases: pair sum in sorted array, merging two sorted arrays, removing duplicates. Then Sliding Window: fixed-size (max sum of subarray of size k) and variable-size (smallest subarray with sum ≥ target). Solve CSES "Subarray Divisibility" and 2–3 CF 1000 problems with these techniques.
Day 6Prefix sums: precompute cumulative sums so any subarray sum is O(1). This is deceptively powerful. Learn 1D prefix sums, then understand the idea behind 2D prefix sums (you don't need to implement 2D yet, just understand it). Solve 2 more prefix sum problems.
Day 7Second virtual contest. Pick a CF Div 3 this time (slightly harder than Div 4). Try to solve A, B, C within the time limit. Upsolve D and E after — even if the editorial feels hard, read it and code the solution from scratch. Log everything in your problem tracker.
💡 Tip: When you see a problem asking about subarrays or contiguous sequences, your first instinct should be: can I use prefix sums? Can I use sliding window? Can I use two pointers? These three techniques alone solve the majority of array problems at the 800–1200 level.
W4

Backtracking & First Live Contest

~3–4 hrs/day

🎯 Goal

Learn complete search and backtracking — the art of generating all possibilities smartly. Practice binary search on answers (a pattern that appears in every other Div 3 contest). Then the big milestone: participate in your first LIVE Codeforces contest. By the end of this week, you should have a CF rating and a clear picture of where you stand.

📅 Day-by-Day

Day 1–2Complete search: generate all permutations (std::next_permutation or recursive), all subsets (bitmask or recursion). Understand when brute force is okay — if N ≤ 20, you can try all 2^N subsets; if N ≤ 10, you can try all N! permutations. Solve CSES "Apple Division" (classic subset brute force). Read CP Handbook Chapter 5.
Day 3Backtracking: the idea is "try a choice, recurse, undo the choice." Classic problems: N-Queens, Sudoku solver, subset sum with pruning. The key skill is identifying when to prune — if the current partial solution can't possibly lead to a valid answer, stop early. Solve CSES "Chessboard and Queens".
Day 4Binary search on answers: this is a pattern where you binary search on the answer itself. "What is the minimum maximum distance?" → binary search on the distance, check if it's feasible. "What is the maximum minimum value?" → same idea. Practice 2–3 problems with this pattern from Striver's binary search section.
Day 5Speed training: solve 5 CF A/B problems in one sitting, aiming for under 10 minutes each. Practice your debugging workflow — when your solution gives WA, test it against a brute force on small random inputs (stress testing). Learn to write a simple stress test script.
Day 6🔴 LIVE CONTEST DAY. Register for the next Codeforces Div 4 (or Div 3) round. During the contest: read all problems first (2 minutes), solve A immediately, then go in order. Don't panic on C/D — read carefully, think for 5 minutes before coding. Submit early and often.
Day 7Post-contest upsolving: this is the single highest-ROI activity in competitive programming. For every problem you didn't solve, read the editorial, understand the approach, then code it from scratch WITHOUT looking at the editorial again. Log each one as "Upsolved" in your tracker. Reflect on the month — what topics felt strong, what needs more work.
💡 Tip: Your first live contest will feel chaotic — that's normal. Your hands will shake, you'll misread a problem, you'll get a WA on something silly. None of that matters. What matters is that you showed up, competed, and upsolve afterward. Every competitive programmer's first contest was a disaster. The second one is always better.
